Design Principles

SOLID Principles

SOLID is an acronym for 5 foundational OOP design principles coined by Robert C. Martin (Uncle Bob). They help you write code that's easy to maintain, extend, and test.

S
Single Responsibility Principle
A class should have one, and only one, reason to change
🧑‍🍳 Real Life

A chef who also does accounting, drives deliveries, and cleans the restaurant is stretched thin. One job per person = better quality. Same for classes.

⚙️ Technical

Each class encapsulates a single concern. When requirements change, only one class should need to update, reducing unintended side effects.

❌ Violates SRPBAD
public class User {
  String name;
  // Job 1: hold data
  public String getName() {...}
  // Job 2: save to DB
  public void save() {...}
  // Job 3: send email
  public void sendEmail() {...}
}
✅ Follows SRPGOOD
// Each class = one job
class User { String name; }

class UserRepository {
  void save(User u) {...}
}

class UserEmailService {
  void sendWelcome(User u) {...}
}
💡If you're naming a class UserManagerHelperService, it has too many responsibilities.
O
Open/Closed Principle
Open for extension, closed for modification
🔌 Real Life

Your phone has a USB-C port. You can plug in new accessories (headphones, chargers) without rebuilding the phone. Add new features without touching working code.

⚙️ Technical

New behavior is added by writing new classes/methods, not by modifying existing tested code. Typically achieved with interfaces and polymorphism.

❌ Violates OCPBAD
class Discount {
  double calc(String type, double p){
    if(type.equals("STUDENT"))
      return p*.8;
    else if(type.equals("SENIOR"))
      return p*.7;
    // Must edit here for new types
    return p;
  }
}
✅ Follows OCPGOOD
interface DiscountPolicy {
  double apply(double price);
}
class StudentDiscount
    implements DiscountPolicy {
  public double apply(double p) {
    return p * 0.8;
  }
}
// New type = new class only!
class VIPDiscount
    implements DiscountPolicy {
  public double apply(double p) {
    return p * 0.5;
  }
}
L
Liskov Substitution Principle
Subtypes must be substitutable for their base types without breaking the program
🦆 Real Life

If a system expects a "Bird that flies", giving it a Penguin shouldn't crash it. If a subclass can't honor the parent's promises, the inheritance design is wrong.

⚙️ Technical

If S is a subtype of T, objects of T may be replaced with objects of S without altering program correctness. Subclasses should never throw unexpected exceptions for inherited methods.

LSPExample.javaJava
// ❌ Bad — Penguin breaks the fly() contract
class Bird { void fly() { /* ok */ } }
class Penguin extends Bird {
  void fly() { throw new UnsupportedOperationException(); }
}

// ✅ Good — model what each bird CAN do
interface Bird        { void eat(); }
interface FlyingBird  extends Bird { void fly(); }

class Eagle   implements FlyingBird {
  public void eat() { System.out.println("Eagle eats"); }
  public void fly() { System.out.println("Eagle soars"); }
}
class Penguin implements Bird {
  public void eat() { System.out.println("Penguin eats"); }
  // No fly() — and that's perfectly correct!
}
I
Interface Segregation Principle
Clients should not be forced to depend on interfaces they don't use
📋 Real Life

Don't hand a janitor a contract requiring them to perform surgery just because both work in a hospital. Give people only the responsibilities relevant to their role.

⚙️ Technical

Break fat interfaces into smaller, focused ones. Implementing classes should only know about methods relevant to them — no empty or exception-throwing implementations.

❌ Fat InterfaceBAD
interface Worker {
  void work();
  void eat();   // Robots can't eat!
  void sleep(); // Robots don't sleep!
}
class Robot implements Worker {
  public void work()  { /* ok */ }
  public void eat()   {
    throw new UnsupportedOperationException();
  }
  public void sleep() {
    throw new UnsupportedOperationException();
  }
}
✅ SegregatedGOOD
interface Workable { void work(); }
interface Eatable  { void eat();  }
interface Sleepable{ void sleep();}

// Robot only implements what fits
class Robot implements Workable {
  public void work() { /* works */ }
}
class Human implements
    Workable, Eatable, Sleepable {
  public void work()  { /* works */ }
  public void eat()   { /* eats  */ }
  public void sleep() { /* sleeps */ }
}
D
Dependency Inversion Principle
Depend on abstractions, not on concrete implementations
💡 Real Life

A lamp plugs into a wall socket — it doesn't care if the power comes from coal or solar. The socket is the abstraction. Swap the power source without rewiring the lamp.

⚙️ Technical

High-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces). This is the backbone of Spring's dependency injection framework.

DIPExample.javaJava
// Abstraction (the "socket")
interface MessageSender { void send(String msg); }

// Low-level implementations
class EmailSender implements MessageSender {
  public void send(String msg) { System.out.println("Email: "+msg); }
}
class SmsSender implements MessageSender {
  public void send(String msg) { System.out.println("SMS: "+msg); }
}

// High-level depends on interface — NOT on EmailSender or SmsSender
class NotificationService {
  private final MessageSender sender;
  // Inject via constructor — swap sender without changing this class!
  public NotificationService(MessageSender sender) {
    this.sender = sender;
  }
  public void notify(String msg) { sender.send(msg); }
}

// Usage
var svc = new NotificationService(new SmsSender());
svc.notify("Hello!"); // SMS: Hello!
💡Spring's @Autowired and @Inject implement DIP automatically — the container injects the right implementation at runtime.

Design Principles

10 Software Design Principles

Beyond SOLID, these principles guide everyday coding decisions to keep codebases clean, predictable, and maintainable.

1

SOLID Principles Foundation

The five rules (SRP, OCP, LSP, ISP, DIP) that make code flexible, maintainable, and scalable. Covered in depth in the section above.

2

DRY — Don't Repeat Yourself

Layman: Don't write the same recipe twice. Technical: Every piece of knowledge should have a single, unambiguous representation in the system. Extract shared logic into methods/classes/utilities.

DRY ExampleJava
// ❌ BAD — calculation duplicated in 3 places
double tax1 = price1 * 0.12;
double tax2 = price2 * 0.12;

// ✅ GOOD — single source of truth
private double calcTax(double price) {
  return price * 0.12;
}
double tax1 = calcTax(price1);
double tax2 = calcTax(price2);
3

KISS — Keep It Simple, Stupid

Layman: Don't use a crane to lift a pencil. Technical: Write the simplest code that solves the problem. Avoid over-engineering. Simple code is easier to read, test, and debug.

⚠️"Any fool can write code a computer understands. Good programmers write code humans can understand." — Martin Fowler
4

YAGNI — You Aren't Gonna Need It

Layman: Don't build a second floor if you only need a garden shed. Technical: Don't add functionality until it's actually needed. Premature abstraction adds complexity without benefit.

YAGNI ExampleBAD
// ❌ BAD — building a "plugin system" when you have 1 feature
interface PluginLoader {...}
class PluginRegistry {...}
class PluginFactory {...}
// Just to log "Hello World"

// ✅ GOOD — do the simplest thing that works NOW
System.out.println("Hello World");
5

Separation of Concerns (SoC)

Layman: Keep the kitchen, dining room, and billing department separate — they have different jobs. Technical: Divide your code into layers/modules where each part handles one specific aspect (UI, business logic, data access).

Controller Layer

Handles HTTP requests, routes, and response mapping only.

Service Layer

Contains business logic and orchestrates operations.

Repository Layer

Handles all database interactions only.

6

Encapsulation

Layman: A TV remote hides all the complex circuitry — you just press buttons. Technical: Keep internal state private; expose only what's necessary through well-defined public methods (getters/setters). Hide implementation details.

Encapsulation.javaJava
public class BankAccount {
  private double balance; // hidden

  public void deposit(double amount) {
    if (amount > 0) balance += amount; // controlled access
  }
  public double getBalance() { return balance; }
  // No direct "balance = -9999" allowed from outside!
}
7

Composition Over Inheritance

Layman: Build a car from parts (engine, wheels, seats) rather than inheriting from a "VehicleWithSeatsAndEngineAndWheels" mega-class. Technical: Prefer combining small, reusable objects over deep inheritance hierarchies. Leads to more flexible, testable designs.

CompositionExample.javaJava
// ❌ Inheritance hell
class FlyingSwimmingDuck extends Duck extends Animal {...}

// ✅ Composition — plug in behaviors
interface FlyBehavior  { void fly(); }
interface SwimBehavior { void swim(); }

class Duck {
  private FlyBehavior  flyBehavior;
  private SwimBehavior swimBehavior;

  public Duck(FlyBehavior fly, SwimBehavior swim) {
    this.flyBehavior  = fly;
    this.swimBehavior = swim;
  }
  public void performFly()  { flyBehavior.fly(); }
  public void performSwim() { swimBehavior.swim(); }
}
8

High Cohesion & Low Coupling

Layman: High Cohesion = keep related things together (all knife tools in the kitchen drawer). Low Coupling = minimize dependencies between modules (kitchen doesn't depend on the bathroom).

🔗 High Cohesion

Methods in a class work together toward a single purpose. A UserService only does user-related things.

✂️ Low Coupling

Classes know as little as possible about each other. Changes in one don't cascade into many others.

9

Law of Demeter (Principle of Least Knowledge)

Layman: Don't talk to strangers. Ask your friend, don't reach into their wallet yourself. Technical: A method should only call methods on: its own class, its parameters, objects it creates, or direct field objects. Avoid chaining like a.getB().getC().doSomething().

❌ BadBAD
// "Train wreck" — too much knowledge
customer
  .getWallet()
  .getMoney()
  .deduct(10);
✅ GoodGOOD
// Customer handles its own payment
customer.pay(10);
// Customer knows how — you don't need to
10

Fail Fast

Layman: If a bridge has a crack, alert everyone immediately — don't wait until it collapses. Technical: Validate inputs early and throw exceptions as soon as something goes wrong. Better to fail loudly at the boundary than silently corrupt data deep inside.

FailFast.javaJava
public void setAge(int age) {
  // Fail fast — validate at the door
  if (age < 0 || age > 150) {
    throw new IllegalArgumentException(
      "Invalid age: " + age);
  }
  this.age = age;
  // Objects.requireNonNull() is another fail-fast tool
}

Creational Pattern

Singleton Creational

🌍 Real Life

There's only one President at a time. Everyone talks to the same person — no one creates a new president each time they need something.

⚙️ Technical

Ensures a class has only one instance and provides a global access point to it. Used for shared resources: DB connections, loggers, config managers.

Singleton.javaJava
public class AppConfig {
  // volatile ensures visibility across threads
  private static volatile AppConfig instance;

  private AppConfig() {} // block "new AppConfig()"

  // Double-checked locking (thread-safe)
  public static AppConfig getInstance() {
    if (instance == null) {
      synchronized (AppConfig.class) {
        if (instance == null)
          instance = new AppConfig();
      }
    }
    return instance;
  }
  public String get(String key) { return "value"; }
}

// Both get SAME object
AppConfig a = AppConfig.getInstance();
AppConfig b = AppConfig.getInstance();
System.out.println(a == b); // true
⚠️Singleton is often considered an anti-pattern when overused — it's a hidden global. Prefer Spring's @Bean (singleton-scoped by default) which is managed and testable.

Creational Pattern

Factory Method Creational

🚗 Real Life

You say "I want a car." A factory decides whether to make a sedan, SUV, or electric car based on your order. You don't know or care how it's built.

⚙️ Technical

Defines an interface for creating an object but lets subclasses/methods decide which class to instantiate. Decouples object creation from its usage.

ShapeFactory.javaJava
interface Shape { void draw(); }

class Circle    implements Shape { public void draw() { System.out.println("⭕ Circle"); } }
class Rectangle implements Shape { public void draw() { System.out.println("🟥 Rectangle"); } }
class Triangle  implements Shape { public void draw() { System.out.println("🔺 Triangle"); } }

class ShapeFactory {
  public static Shape create(String type) {
    return switch(type.toLowerCase()) {
      case "circle"    -> new Circle();
      case "rectangle" -> new Rectangle();
      case "triangle"  -> new Triangle();
      default -> throw new IllegalArgumentException("Unknown: " + type);
    };
  }
}

// Usage — caller doesn't know the concrete class
Shape s = ShapeFactory.create("circle");
s.draw(); // ⭕ Circle

Creational Pattern

Abstract Factory Creational

🛋️ Real Life

IKEA and Ashley are furniture factories. Each makes matching sets (chair + table + sofa) in their own style. You pick a factory and everything you get looks coordinated.

⚙️ Technical

A factory of factories. Creates families of related objects without specifying their concrete classes. Ensures product objects from the same family are compatible.

AbstractFactory.javaJava
interface Button { void render(); }
interface TextField { void render(); }

// Abstract factory
interface UIFactory {
  Button    createButton();
  TextField createTextField();
}

// Windows family
class WindowsButton    implements Button    { public void render() { System.out.println("Windows Button"); }}
class WindowsTextField implements TextField { public void render() { System.out.println("Windows TextField"); }}
class WindowsFactory implements UIFactory {
  public Button    createButton()    { return new WindowsButton(); }
  public TextField createTextField() { return new WindowsTextField(); }
}

// Mac family (same structure, different style)
class MacFactory implements UIFactory {
  public Button    createButton()    { /* Mac Button */ return () -> System.out.println("Mac Button"); }
  public TextField createTextField() { /* Mac TextField */ return () -> System.out.println("Mac TextField"); }
}

// Usage — swap whole UI family by changing factory
UIFactory factory = new WindowsFactory(); // or MacFactory
factory.createButton().render();
factory.createTextField().render();

Creational Pattern

Builder Creational

🍔 Real Life

Ordering a custom burger: "beef patty, extra cheese, no onions, add bacon." Build it step-by-step. No need for a 10-parameter constructor like Burger(beef, cheese, null, true, false...).

⚙️ Technical

Separates the construction of a complex object from its representation. Allows the same construction process to create different representations using a fluent API.

Person.java (Builder)Java
public class Person {
  private final String firstName, lastName, email, phone;
  private final int age;

  private Person(Builder b) {
    firstName=b.firstName; lastName=b.lastName;
    age=b.age; email=b.email; phone=b.phone;
  }

  public static class Builder {
    private String firstName, lastName, email, phone;
    private int age;

    public Builder firstName(String v) { firstName=v; return this; }
    public Builder lastName(String v)  { lastName=v;  return this; }
    public Builder age(int v)         { age=v;       return this; }
    public Builder email(String v)    { email=v;     return this; }
    public Builder phone(String v)    { phone=v;     return this; }
    public Person  build()            { return new Person(this); }
  }
}

// Crystal-clear, no mysterious argument positions!
Person p = new Person.Builder()
  .firstName("Maria").lastName("Santos")
  .age(28).email("[email protected]")
  .build();
💡Lombok's @Builder annotation auto-generates all this boilerplate for you!

Creational Pattern

Prototype Creational

🐑 Real Life

Dolly the cloned sheep. Instead of creating a new sheep from scratch (going through expensive development), you clone an existing one. Same for expensive-to-create objects.

⚙️ Technical

Create new objects by copying (cloning) an existing object (the prototype). Avoids expensive initialization when creating multiple similar objects. Java's Cloneable interface supports this.

Prototype.javaJava
public class DatabaseConfig implements Cloneable {
  private String host;
  private int port;
  private String dbName;

  public DatabaseConfig(String host, int port, String db) {
    this.host=host; this.port=port; this.dbName=db;
  }

  @Override
  public DatabaseConfig clone() {
    try {
      return (DatabaseConfig) super.clone();
    } catch (CloneNotSupportedException e) {
      throw new RuntimeException(e);
    }
  }
}

// Create one expensive config, clone for variations
DatabaseConfig master = new DatabaseConfig("prod-server", 5432, "maindb");
DatabaseConfig replica = master.clone();
// replica.host = "replica-server"; // tweak only what differs

Structural Pattern

Adapter Structural

✈️ Real Life

Traveling to Japan with a US plug? You use a travel adapter — it doesn't change the device or the wall, it just bridges the gap between two incompatible interfaces.

⚙️ Technical

Converts the interface of a class into another interface that clients expect. Enables classes with incompatible interfaces to work together. Common when integrating third-party libraries.

Adapter.javaJava
// Our app's expected interface
interface MediaPlayer { void play(String file); }

// Third-party library we cannot change
class VlcPlayer {
  public void playVlc(String file) {
    System.out.println("VLC: " + file);
  }
}

// Adapter — wraps VlcPlayer to match MediaPlayer
class VlcAdapter implements MediaPlayer {
  private final VlcPlayer vlc = new VlcPlayer();
  public void play(String file) { vlc.playVlc(file); }
}

// Our app talks to MediaPlayer only
MediaPlayer player = new VlcAdapter();
player.play("movie.vlc");

Structural Pattern

Bridge Structural

📺 Real Life

A TV remote (abstraction) controls different TV brands (implementations). You separate the remote's design from which TV brand it talks to — both can evolve independently.

⚙️ Technical

Decouples an abstraction from its implementation so the two can vary independently. Avoids a cartesian product of subclasses (e.g. RedCircle, BlueCircle, RedSquare, BlueSquare → just Circle+Red/Blue).

Bridge.javaJava
// Implementation interface
interface Renderer { void render(String shape); }
class VectorRenderer implements Renderer {
  public void render(String shape) {
    System.out.println("Vector: " + shape);
  }
}
class RasterRenderer implements Renderer {
  public void render(String shape) {
    System.out.println("Raster pixels: " + shape);
  }
}

// Abstraction — holds a reference to implementation
abstract class Shape {
  protected Renderer renderer;
  Shape(Renderer r) { this.renderer = r; }
  abstract void draw();
}
class Circle extends Shape {
  Circle(Renderer r) { super(r); }
  public void draw() { renderer.render("Circle"); }
}

// Mix any shape with any renderer!
new Circle(new VectorRenderer()).draw(); // Vector: Circle
new Circle(new RasterRenderer()).draw(); // Raster pixels: Circle

Structural Pattern

Composite Structural

📁 Real Life

A file system: a folder can contain files OR other folders. Whether you call "size()" on a single file or a folder full of files, it just works — the caller doesn't care which it is.

⚙️ Technical

Composes objects into tree structures. Treats individual objects and compositions uniformly through a common interface. Used in file systems, UI component trees, org charts.

Composite.javaJava
interface FileSystemItem {
  void show(String indent);
}

class File implements FileSystemItem {
  private String name;
  File(String n) { name=n; }
  public void show(String indent) { System.out.println(indent + "📄 " + name); }
}

class Folder implements FileSystemItem {
  private String name;
  private List<FileSystemItem> items = new ArrayList<>();
  Folder(String n) { name=n; }
  public void add(FileSystemItem i) { items.add(i); }
  public void show(String indent) {
    System.out.println(indent + "📁 " + name);
    items.forEach(i -> i.show(indent + "  "));
  }
}

// Usage
Folder root = new Folder("root");
Folder src  = new Folder("src");
src.add(new File("Main.java"));
src.add(new File("App.java"));
root.add(src);
root.add(new File("README.md"));
root.show("");

Structural Pattern

Decorator Structural

☕ Real Life

Start with plain coffee. Add milk → add sugar → add whipped cream. Each addition wraps the previous. The result is still a coffee, just richer — no need for a MilkSugarWhipCoffee class.

⚙️ Technical

Attaches additional responsibilities to an object dynamically. An alternative to subclassing for extending functionality. Java's InputStream → BufferedInputStream → DataInputStream is a real-world example.

CoffeeDecorator.javaJava
interface Coffee { String getDesc(); double cost(); }

class SimpleCoffee implements Coffee {
  public String getDesc() { return "Coffee"; }
  public double cost()    { return 1.0; }
}

abstract class CoffeeDecorator implements Coffee {
  protected Coffee wrapped;
  CoffeeDecorator(Coffee c) { wrapped=c; }
}

class Milk extends CoffeeDecorator {
  Milk(Coffee c) { super(c); }
  public String getDesc() { return wrapped.getDesc() + ", Milk"; }
  public double cost()    { return wrapped.cost() + 0.25; }
}
class Whip extends CoffeeDecorator {
  Whip(Coffee c) { super(c); }
  public String getDesc() { return wrapped.getDesc() + ", Whip"; }
  public double cost()    { return wrapped.cost() + 0.50; }
}

Coffee order = new Whip(new Milk(new SimpleCoffee()));
System.out.println(order.getDesc()); // Coffee, Milk, Whip
System.out.println(order.cost());    // 1.75

Structural Pattern

Facade Structural

🎬 Real Life

Watching a movie: you press "Watch Movie" and the TV turns on, the lights dim, the surround sound activates. One button, hidden complexity. The button is the Facade.

⚙️ Technical

Provides a simplified interface to a complex subsystem. Reduces dependencies between client code and multiple subsystem classes. Does not prevent advanced users from accessing the subsystem directly.

HomeTheaterFacade.javaJava
class TV     { void on() {System.out.println("TV on");} }
class Lights { void dim() {System.out.println("Lights dimmed");} }
class Sound  { void on() {System.out.println("Surround on");} }

// Facade — one simple interface to the whole subsystem
class HomeTheaterFacade {
  private TV tv = new TV();
  private Lights lights = new Lights();
  private Sound sound = new Sound();

  public void watchMovie() {
    System.out.println("--- Movie Mode ---");
    lights.dim();
    tv.on();
    sound.on();
  }
}

// Client calls one method — all complexity hidden
new HomeTheaterFacade().watchMovie();

Structural Pattern

Flyweight Structural

🌲 Real Life

A forest with 10,000 trees: instead of storing color, texture, and shape for each tree separately, share one "Oak tree blueprint" and only store each tree's x,y position separately.

⚙️ Technical

Uses sharing to efficiently support large numbers of fine-grained objects. Separates intrinsic state (shared) from extrinsic state (unique per instance). Drastically reduces memory use.

Flyweight.javaJava
// Flyweight — shared intrinsic state
class TreeType {
  private String name, color, texture;
  TreeType(String n, String c, String t) { name=n; color=c; texture=t; }
  void draw(int x, int y) {  // x,y = extrinsic (unique per tree)
    System.out.println(color + " " + name + " at ("+x+","+y+")");
  }
}

// Flyweight Factory — cache and reuse TreeType objects
class TreeFactory {
  private static Map<String,TreeType> cache = new HashMap<>();

  static TreeType get(String name, String color, String texture) {
    String key = name + color + texture;
    return cache.computeIfAbsent(key, k -> new TreeType(name, color, texture));
  }
}

// 10,000 trees share only a few TreeType objects!
TreeType oak = TreeFactory.get("Oak", "Green", "Rough");
oak.draw(10, 20);  // same object, different positions
oak.draw(50, 80);

Structural Pattern

Proxy Structural

🔐 Real Life

A celebrity's agent acts as a proxy. You can't contact the celebrity directly — you go through the agent who handles scheduling, validates requests, and forwards only legitimate ones.

⚙️ Technical

Provides a surrogate or placeholder for another object. Controls access to the real object. Types: Virtual (lazy init), Protection (access control), Remote (different address space), Caching proxy.

Proxy.javaJava
interface Image { void display(); }

class RealImage implements Image {
  private String filename;
  RealImage(String f) {
    filename = f;
    System.out.println("Loading " + f + " from disk..."); // expensive!
  }
  public void display() { System.out.println("Displaying: " + filename); }
}

// Virtual Proxy — lazy load expensive object
class ImageProxy implements Image {
  private String filename;
  private RealImage realImage; // null until needed

  ImageProxy(String f) { filename = f; }

  public void display() {
    if (realImage == null)
      realImage = new RealImage(filename); // load only on first use
    realImage.display();
  }
}

Image img = new ImageProxy("photo.jpg");
// No disk load yet...
img.display(); // Now it loads + displays
img.display(); // Already loaded — just displays
💡Spring AOP (aspect-oriented programming) uses proxies to add logging, transactions, and security transparently.

Behavioral Pattern

Chain of Responsibility Behavioral

🏥 Real Life

Customer support escalation: first the chatbot tries, then Level 1 support, then Level 2, then the manager. Each handler decides to handle it or pass it up the chain.

⚙️ Technical

Passes a request along a chain of handlers. Each handler decides to process the request or pass it to the next. Decouples sender from receiver. Used in middleware, filter pipelines, event handling.

ChainOfResponsibility.javaJava
abstract class SupportHandler {
  protected SupportHandler next;
  public SupportHandler setNext(SupportHandler h) { next=h; return h; }
  public abstract void handle(int level);
}

class Level1Support extends SupportHandler {
  public void handle(int level) {
    if (level <= 1) System.out.println("L1 resolved issue");
    else if (next != null) next.handle(level);
  }
}
class Level2Support extends SupportHandler {
  public void handle(int level) {
    if (level <= 2) System.out.println("L2 resolved issue");
    else if (next != null) next.handle(level);
  }
}
class ManagerSupport extends SupportHandler {
  public void handle(int level) { System.out.println("Manager resolved issue"); }
}

// Build chain: L1 → L2 → Manager
SupportHandler l1 = new Level1Support();
l1.setNext(new Level2Support()).setNext(new ManagerSupport());
l1.handle(3); // Manager resolved issue

Behavioral Pattern

Command Behavioral

↩️ Real Life

The "Undo" button in a text editor. Each action (type, delete, paste) is wrapped as a Command object that knows how to execute AND undo itself. Store them in a history stack.

⚙️ Technical

Encapsulates a request as an object, allowing you to queue, log, and undo operations. Decouples the sender from the receiver of the request.

Command.javaJava
interface Command { void execute(); void undo(); }

class TextEditor {
  private StringBuilder text = new StringBuilder();
  void append(String s) { text.append(s); System.out.println("Text: "+text); }
  void deleteLast(int n) { text.delete(text.length()-n, text.length()); System.out.println("Text: "+text); }
}

class TypeCommand implements Command {
  private TextEditor editor; private String typed;
  TypeCommand(TextEditor e, String t) { editor=e; typed=t; }
  public void execute() { editor.append(typed); }
  public void undo()    { editor.deleteLast(typed.length()); }
}

// History stack for undo
Deque<Command> history = new ArrayDeque<>();
TextEditor editor = new TextEditor();
Command cmd = new TypeCommand(editor, "Hello");
cmd.execute(); history.push(cmd); // Text: Hello
history.pop().undo();             // Text: (empty)

Interpreter Behavioral

🌐 Real Life

A translator who speaks both English and Spanish. Given a sentence in one language, they interpret and produce the meaning in the other. The interpreter has rules for reading and translating.

⚙️ Technical

Defines a grammar for a language and provides an interpreter to deal with that grammar. Used in SQL parsers, regular expressions, calculator evaluators, scripting engines.

Interpreter.java (simple math)Java
interface Expression { int interpret(); }

class Number implements Expression {
  private int val;
  Number(int v) { val=v; }
  public int interpret() { return val; }
}
class Add implements Expression {
  private Expression left, right;
  Add(Expression l, Expression r) { left=l; right=r; }
  public int interpret() { return left.interpret() + right.interpret(); }
}
class Multiply implements Expression {
  private Expression left, right;
  Multiply(Expression l, Expression r) { left=l; right=r; }
  public int interpret() { return left.interpret() * right.interpret(); }
}

// Interpret: (3 + 4) * 2 = 14
Expression expr = new Multiply(
  new Add(new Number(3), new Number(4)),
  new Number(2)
);
System.out.println(expr.interpret()); // 14

Iterator Behavioral

📖 Real Life

A TV remote's "next channel" button. You don't need to know how channels are stored internally — you just press next/prev to traverse them one by one.

⚙️ Technical

Provides a way to sequentially access elements of a collection without exposing its underlying structure. Java's Iterator<T>, for-each, and Iterable<T> implement this pattern natively.

Iterator.javaJava
// Java's Iterator is already built in
import java.util.*;

class BookCollection implements Iterable<String> {
  private List<String> books = new ArrayList<>();
  public void add(String book) { books.add(book); }

  @Override
  public Iterator<String> iterator() {
    return new Iterator<String>() {
      private int index = 0;
      public boolean hasNext() { return index < books.size(); }
      public String next()     { return books.get(index++); }
    };
  }
}

BookCollection shelf = new BookCollection();
shelf.add("Clean Code");
shelf.add("Design Patterns");
shelf.add("Refactoring");

// for-each works because we implement Iterable
for (String book : shelf) {
  System.out.println("📗 " + book);
}

Mediator Behavioral

✈️ Real Life

Air traffic control. Planes don't communicate directly with each other — they all talk to the control tower (mediator) which coordinates everything. No direct plane-to-plane chaos.

⚙️ Technical

Defines an object that encapsulates how a set of objects interact. Promotes loose coupling by keeping objects from referring to each other explicitly. Used in chat rooms, event buses, MVC controllers.

Mediator.javaJava
interface ChatMediator { void sendMessage(String msg, User from); }

class ChatRoom implements ChatMediator {
  private List<User> users = new ArrayList<>();
  void join(User u) { users.add(u); }
  public void sendMessage(String msg, User from) {
    users.stream()
      .filter(u -> u != from)
      .forEach(u -> u.receive(from.name + ": " + msg));
  }
}

class User {
  String name; private ChatMediator mediator;
  User(String n, ChatMediator m) { name=n; mediator=m; }
  void send(String msg)    { mediator.sendMessage(msg, this); }
  void receive(String msg) { System.out.println("[📨 " + name + "] " + msg); }
}

ChatRoom room = new ChatRoom();
User alice = new User("Alice", room);
User bob   = new User("Bob", room);
room.join(alice); room.join(bob);
alice.send("Hey Bob!"); // [📨 Bob] Alice: Hey Bob!

Memento Behavioral

🎮 Real Life

A video game save point. You can save your game state and restore it later without knowing how the game engine stores things internally. Snapshot and rewind.

⚙️ Technical

Captures and externalizes an object's internal state without violating encapsulation, so it can be restored later. The "originator" creates mementos, the "caretaker" stores them.

Memento.javaJava
// Memento — snapshot of state
class GameMemento {
  private final int level, health;
  GameMemento(int l, int h) { level=l; health=h; }
  int getLevel()  { return level; }
  int getHealth() { return health; }
}

// Originator — creates and restores from memento
class Game {
  private int level, health;
  Game(int l, int h) { level=l; health=h; }
  void play() { level++; health-=20; System.out.println("Level:"+level+" HP:"+health); }
  GameMemento save() { return new GameMemento(level, health); }
  void restore(GameMemento m) { level=m.getLevel(); health=m.getHealth(); System.out.println("Restored! Level:"+level+" HP:"+health); }
}

Game g = new Game(1, 100);
GameMemento save = g.save(); // save at Level:1 HP:100
g.play(); g.play();           // Level:3 HP:60
g.restore(save);             // back to Level:1 HP:100

Observer Behavioral

📺 Real Life

YouTube subscriptions. When a creator uploads a video, all subscribers are notified automatically. The channel doesn't know who's subscribed — it just broadcasts; listeners react.

⚙️ Technical

Defines a one-to-many dependency. When one object (Subject) changes state, all its dependents (Observers) are notified automatically. Foundation of event-driven programming, MVC, and reactive frameworks.

Observer.javaJava
interface Observer { void update(String event); }

class EventBus {
  private Map<String, List<Observer>> listeners = new HashMap<>();

  void subscribe(String event, Observer o) {
    listeners.computeIfAbsent(event, k -> new ArrayList<>()).add(o);
  }

  void publish(String event) {
    listeners.getOrDefault(event, Collections.emptyList())
      .forEach(o -> o.update(event));
  }
}

// Usage
EventBus bus = new EventBus();
bus.subscribe("ORDER_PLACED", e -> System.out.println("📧 Sending confirmation email"));
bus.subscribe("ORDER_PLACED", e -> System.out.println("📦 Starting fulfillment"));
bus.subscribe("ORDER_PLACED", e -> System.out.println("📊 Updating analytics"));

bus.publish("ORDER_PLACED"); // All 3 fire!
💡Spring's ApplicationEventPublisher / @EventListener and reactive frameworks like RxJava and Project Reactor are built on this pattern.

Behavioral Pattern

State Behavioral

🚦 Real Life

A traffic light: same intersection behaves differently (stop, go, slow) based on its current state. The light itself doesn't change — its state changes and dictates behavior.

⚙️ Technical

Allows an object to alter its behavior when its internal state changes — it will appear to change its class. Eliminates large if/switch chains based on state flags. Each state is a separate class.

State.javaJava
interface OrderState { void handle(Order order); }

class PendingState  implements OrderState {
  public void handle(Order o) {
    System.out.println("Processing payment...");
    o.setState(new ShippedState());
  }
}
class ShippedState  implements OrderState {
  public void handle(Order o) {
    System.out.println("Package shipped! 📦");
    o.setState(new DeliveredState());
  }
}
class DeliveredState implements OrderState {
  public void handle(Order o) { System.out.println("Delivered! ✅"); }
}

class Order {
  private OrderState state = new PendingState();
  void setState(OrderState s) { state = s; }
  void next() { state.handle(this); }
}

Order o = new Order();
o.next(); // Processing payment...
o.next(); // Package shipped!
o.next(); // Delivered!

Strategy Behavioral

🗺️ Real Life

Getting from A to B: drive, take the bus, or cycle. Same destination, different strategy. You choose at runtime and swap strategies without changing the destination logic.

⚙️ Technical

Defines a family of algorithms, encapsulates each one, and makes them interchangeable. The client chooses which algorithm to use at runtime. Follows OCP perfectly.

Strategy.javaJava
interface PaymentStrategy { void pay(double amount); }

class CreditCard implements PaymentStrategy {
  public void pay(double amt) { System.out.println("💳 Charged $"+amt+" to card"); }
}
class GCash implements PaymentStrategy {
  public void pay(double amt) { System.out.println("📱 Sent $"+amt+" via GCash"); }
}
class Cash implements PaymentStrategy {
  public void pay(double amt) { System.out.println("💵 Paid $"+amt+" cash"); }
}

class Checkout {
  private PaymentStrategy strategy;
  void setStrategy(PaymentStrategy s) { strategy = s; }
  void processOrder(double total) { strategy.pay(total); }
}

Checkout cart = new Checkout();
cart.setStrategy(new GCash());
cart.processOrder(499.00); // 📱 Sent $499.0 via GCash

Template Method Behavioral

☕ Real Life

Making a hot drink: boil water → brew → pour → add extras. The skeleton is identical for tea and coffee; only the "brew" and "extras" steps differ. Fix the template, vary the details.

⚙️ Technical

Defines the skeleton of an algorithm in a base class, deferring some steps to subclasses. Subclasses can redefine certain steps without changing the algorithm's overall structure. The template method is usually declared final.

TemplateMethod.javaJava
abstract class DataReport {
  // Template method — final = order cannot change
  public final void generate() {
    fetchData();     // common step
    processData();   // subclass fills in
    formatOutput(); // subclass fills in
    export();        // common step
  }
  private void fetchData() { System.out.println("Fetching from DB..."); }
  private void export()    { System.out.println("Exporting file..."); }

  protected abstract void processData();
  protected abstract void formatOutput();
}

class PDFReport extends DataReport {
  protected void processData()  { System.out.println("Processing for PDF"); }
  protected void formatOutput() { System.out.println("Formatting as PDF 📄"); }
}
class ExcelReport extends DataReport {
  protected void processData()  { System.out.println("Processing for Excel"); }
  protected void formatOutput() { System.out.println("Formatting as XLSX 📊"); }
}

new PDFReport().generate();

Visitor Behavioral

🏥 Real Life

A health inspector visits different types of businesses (restaurant, pharmacy, gym) and performs a different inspection procedure for each — without the businesses knowing the inspection details.

⚙️ Technical

Lets you add further operations to objects without modifying them. Separates an algorithm from the objects it operates on. Useful for adding operations to object hierarchies (AST traversal, DOM processing).

Visitor.javaJava
interface Visitor {
  void visit(Circle circle);
  void visit(Rectangle rect);
}
interface Shape { void accept(Visitor v); }

class Circle implements Shape {
  double radius;
  Circle(double r) { radius=r; }
  public void accept(Visitor v) { v.visit(this); }
}
class Rectangle implements Shape {
  double w, h;
  Rectangle(double w, double h) { this.w=w; this.h=h; }
  public void accept(Visitor v) { v.visit(this); }
}

// Add "area calculation" without touching Shape classes
class AreaCalculator implements Visitor {
  public void visit(Circle c)    { System.out.println("Circle area: " + (Math.PI*c.radius*c.radius)); }
  public void visit(Rectangle r) { System.out.println("Rect area: " + (r.w*r.h)); }
}

List<Shape> shapes = List.of(new Circle(5), new Rectangle(4,6));
Visitor calc = new AreaCalculator();
shapes.forEach(s -> s.accept(calc));

⬡ ACID Properties
🧪
Database Fundamentals

ACID Properties

ACID properties define the guarantees that database transactions must provide to ensure data integrity, especially in concurrent and failure scenarios.

A

Atomicity — "All or Nothing"

Layman: A bank transfer either FULLY succeeds (deduct from A AND add to B) or completely fails — you never lose money that "disappeared" mid-transfer.

Technical: A transaction is treated as a single unit. Either all operations in it succeed, or none are committed. If any step fails, the entire transaction is rolled back.

Atomicity (Spring)Java
@Transactional // ← ensures atomicity
public void transferMoney(Account from, Account to, double amount) {
  from.debit(amount);   // Step 1
  to.credit(amount);    // Step 2
  // If Step 2 fails → Step 1 is ROLLED BACK automatically
}
C

Consistency — "Valid State to Valid State"

Layman: Your checking account can never go below $0 (if that's the rule). The database enforces the rules — before and after every transaction, all rules hold.

Technical: A transaction brings the database from one valid state to another valid state, respecting all defined rules (constraints, cascades, triggers). If a transaction would violate a constraint, it's rejected entirely.

💡Consistency is enforced via DB constraints: NOT NULL, UNIQUE, CHECK, FOREIGN KEY, and application-level validation.
I

Isolation — "Transactions Don't See Each Other's Mess"

Layman: Two cashiers process different orders at the same time. They shouldn't interfere — one cashier's half-processed order shouldn't be visible to the other.

Technical: Concurrent transactions execute as if they were serial. Isolation prevents dirty reads (seeing uncommitted data), non-repeatable reads, and phantom reads.

Isolation LevelsJava
@Transactional(isolation = Isolation.READ_COMMITTED)
public void updateInventory() { /* only reads committed data */ }

// Isolation Levels (weakest → strongest):
// READ_UNCOMMITTED → dirty reads possible (fastest)
// READ_COMMITTED   → no dirty reads (PostgreSQL default)
// REPEATABLE_READ  → same read yields same result within tx
// SERIALIZABLE     → fully isolated, slowest but safest
D

Durability — "Once Committed, Always Committed"

Layman: Once the ATM prints "Transaction Successful" — even if the power goes out 1 second later, your money is safe. The change is permanent.

Technical: Once a transaction is committed, it persists even through crashes, power failures, or errors. Implemented via write-ahead logging (WAL) and disk persistence in database engines.

💡Durability is a DB engine guarantee (MySQL, PostgreSQL, etc.). As a developer, ensure you always commit or roll back transactions and never rely on in-memory state for critical data.
⬡ Java Best Practices

Best Practices

Java Best Practices

Practices that every professional Java developer should follow to write efficient, safe, and maintainable code.

1
Prefer Immutability
Immutable objects are thread-safe and predictable by nature
Immutability.javaJava
// ✅ Immutable class: final class, final fields, no setters
public final class Money {
  private final double amount;
  private final String currency;

  public Money(double amount, String currency) {
    this.amount = amount;
    this.currency = currency;
  }
  public double getAmount()   { return amount; }
  public String getCurrency() { return currency; }

  // Return new object instead of mutating
  public Money add(Money other) {
    return new Money(this.amount + other.amount, currency);
  }
}
// Java 14+ records are immutable by default:
record Point(int x, int y) {}
2
Use Optional Instead of Null
Avoid NullPointerException — the "billion dollar mistake"
Optional.javaJava
// ❌ BAD — null-prone
public User findById(int id) {
  return db.find(id); // might return null
}
User user = findById(1);
System.out.println(user.getName()); // NullPointerException!

// ✅ GOOD — explicit about possible absence
public Optional<User> findById(int id) {
  return Optional.ofNullable(db.find(id));
}

Optional<User> user = findById(1);
user.map(User::getName)
    .ifPresent(name -> System.out.println("Hello, " + name));

user.orElse(User.guest());          // default value
user.orElseThrow(UserNotFoundException::new); // throw if absent
3
Use Streams & Lambdas (Functional Style)
Declarative code is more readable than imperative loops
❌ ImperativeOLD WAY
List<String> result = new ArrayList<>();
for (User u : users) {
  if (u.isActive() && u.getAge() >= 18) {
    result.add(u.getName().toUpperCase());
  }
}
Collections.sort(result);
✅ DeclarativeMODERN
List<String> result = users.stream() .filter(User::isActive) .filter(u -> u.getAge() >= 18) .map(User::getName) .map(String::toUpperCase) .sorted() .collect(Collectors.toList());
4
Proper Exception Handling
Be specific, don't catch Exception, create domain exceptions
Exceptions.javaJava
// ❌ BAD — catching too broadly
try { doSomething(); } catch (Exception e) { e.printStackTrace(); }

try {
  orderService.process(orderId);

  throw new ResponseStatusException(HttpStatus.NOT_FOUND, e.getMessage());
} catch (PaymentFailedException e) {

// ✅ Custom domain exception
public class OrderNotFoundException extends RuntimeException {
  public OrderNotFoundException(long id) {

  }
}

Write Testable Code (Unit Testing)
Code that's easy to test is usually well-designed code
UserServiceTest.java (JUnit 5)Java
import org.junit.jupiter.api.*;
import org.mockito.*;

class UserServiceTest {
  @Mock  UserRepository mockRepo;
  @InjectMocks UserService userService;

  @BeforeEach
  void setup() { MockitoAnnotations.openMocks(this); }

  @Test
  @DisplayName("should return user when valid ID provided")

    // Arrange
    User expected = new User(1, "Maria");

    // Act

    // Assert
    Assertions.assertEquals("Maria", result.getName());

  @Test

    Mockito.when(mockRepo.findById(99)).thenReturn(Optional.empty());

  }
}

Resource Management (try-with-resources)
Always close resources to prevent memory leaks
❌ Resource LeakBAD
